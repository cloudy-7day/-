# 论文总结与每日更新恢复机制设计

## 背景与根因

当前每日更新任务会在未取得 `DEEPSEEK_API_KEY` 时静默使用通用兜底文案。发布前校验只要求摘要非空，因此 GitHub Actions 即使没有生成真实总结也会显示成功，并把原有高质量内容覆盖为占位内容。

当前工作流还把“当天归档文件存在”视为更新完成。这个判断无法发现归档内容仍是昨天的数据、发布文件未同步、7 篇链接完全未变化或论文仍停留在临时总结的情况。代码推送和手动触发又会无条件重跑，可能重复覆盖当天已经合格的数据。

## 目标

- 尽量在洛杉矶时间 08:07 完成当天 7 篇内容的首次发布。
- 在 09:07 进行独立恢复检查；如果实际内容没有更新，则重新生成。
- DeepSeek 临时不可用时，论文显示基于公开全文的明确标注摘录，不再显示通用占位文案。
- DeepSeek 恢复后自动把临时摘录升级为正式总结，且不重新选择当天文章。
- 任何失败都不得破坏上一份可用数据；工作流状态必须反映真实结果。

## 非目标

- 不接入 GPT 或第二家模型服务。
- 不改变每日 3 篇国际新闻、2 篇 AI 文章和最多 2 篇应用型论文的选择规则。
- 不绕过订阅墙、登录、验证码或反爬限制。
- 不重构前端视觉设计或迁移现有 JSON 存储。

## 数据模型

每篇内容新增 `summarySource`：

- `deepseek`：DeepSeek 基于允许访问的内容生成正式总结。
- `source_extract`：DeepSeek 不可用时，从公开来源内容生成可追溯的临时摘录。

论文的 `source_extract` 必须来自已通过正文可读性门槛的 PDF 文本。临时中文展示使用固定说明加原文摘录，不伪装成中文 AI 总结；英文展示保留原文摘录。通用的 `Local fallback` 和“智能总结需要 DeepSeek key”不得再进入发布数据。

每日根对象新增 `updateStatus`：

- `complete`：7 篇内容均已发布，所有总结来自 DeepSeek。
- `degraded`：内容已更新，但至少一篇使用可追溯的来源摘录。

同时保存稳定的 `contentFingerprint`。指纹由当天 7 篇规范化 URL 按固定顺序计算，只用于判断整批内容是否与前一天完全相同，不会因为摘要文字变化而改变。

## 组件与数据流

### 1. 候选收集

现有新闻、AI 和论文收集流程保持不变。论文仍须有公开全文，并通过 PDF 文本长度、方法信号和应用信号检查。

### 2. 总结生成

每篇内容先调用 DeepSeek。请求遇到超时、限流或临时服务错误时进行有限次数的退避重试。密钥缺失或重试耗尽后：

- 新闻和 AI 内容使用公开 RSS/API 描述形成来源摘录，不能使用与文章无关的通用占位句。
- 论文从已提取正文中优先选择摘要、方法和结论附近的有效句子，生成可追溯的原文摘录。
- 条目标记为 `summarySource: source_extract`，每日状态标记为 `degraded`。

### 3. 发布

候选结果先在内存中完成校验，再写入临时文件。只有日期、数量、分类、唯一链接、翻译字段、论文卡片、总结来源和内容指纹全部有效时，才替换 `data/articles.json` 和当天归档。任何校验失败都保留上一份发布数据。

当天归档已经为 `complete` 时，普通定时、代码推送和非强制手动运行均跳过，防止重复覆盖。强制手动更新必须通过明确输入启用。

### 4. 临时总结升级

当天归档为 `degraded` 时，恢复任务读取已选定的 7 篇内容，只重新执行总结步骤。任务可以从这些条目已经记录的公开 URL 重新取得 RSS/API 描述或论文 PDF 正文，但不得重新运行候选排名或替换文章。DeepSeek 成功后更新总结、论文卡片翻译和状态；全部升级成功后，根状态改为 `complete`。

## 08:07 与 09:07 调度

GitHub Actions 同时配置覆盖夏令时和冬令时的 UTC 触发点。任务开始后使用 `America/Los_Angeles` 计算本地日期和时间：

- 本地时间早于 08:00 时直接跳过，避免冬令时组合导致 07:00 提前发布。
- 08:00 后的首次有效触发执行当天生成；GitHub 排队可能造成分钟级延迟，因此目标为约 08:07，而不是保证 08:00:00。
- 09:00 后的恢复触发执行实际内容检查，目标为约 09:07。

09:00 检查不得只判断归档文件是否存在。它必须验证：

1. `data/articles.json.issueDate` 是洛杉矶当天日期。
2. 当天数据恰好包含 7 篇且分类组合有效。
3. 当前发布文件与当天归档内容一致。
4. 当天 `contentFingerprint` 不与前一天完全相同。
5. 当天是否仍为 `degraded`。

处理规则：

- 当天数据缺失、仍是昨天日期、结构无效、发布文件未同步或内容指纹与前一天完全相同：重新收集并生成整批内容。
- 内容已经更新但状态为 `degraded`：只升级总结。
- 内容有效且状态为 `complete`：幂等跳过。

## 错误处理与可观察性

- 日志明确区分 `fresh_generation`、`summary_upgrade`、`repair_publish` 和 `already_complete`。
- 日志记录 DeepSeek 密钥是否存在，但绝不输出密钥值。
- 每个外部来源的失败只记录来源、阶段和错误摘要，不记录受保护内容。
- 工作流结尾输出当天日期、文章数、指纹、正式总结数、临时摘录数和最终动作。
- 无法生成合格来源摘录或无法通过发布校验时，任务失败并保留旧数据。

## 前端展示

正式总结保持现有样式。`summarySource: source_extract` 时，在摘要上方显示清晰标签：“DeepSeek 暂不可用，当前为公开原文自动摘录”。标签随中英文模式切换。旧归档没有 `summarySource` 时继续按现有方式展示，保证向后兼容。

## 测试设计

按测试先行实现以下回归场景：

1. 缺少 DeepSeek 密钥时，论文产生真实来源摘录并标记 `source_extract`。
2. API 临时失败时按规定重试，成功后标记 `deepseek`。
3. 重试耗尽时不得生成通用占位文案。
4. 完整的当天归档会幂等跳过。
5. `degraded` 当天归档只升级总结，不改变文章 URL 指纹。
6. 09:00 检查发现日期陈旧、结构无效、发布未同步或整批 URL 与前一天完全相同时，会重新生成。
7. 发布校验失败时，现有 `data/articles.json` 和归档不变。
8. 前端正确显示临时摘录标签，旧归档仍可正常打开。
9. 现有论文筛选、AI 筛选、翻译、站点契约和前端测试继续通过。

## 验收标准

- 工作流在洛杉矶本地时间 08:00 前不会生成当天内容，并在约 08:07 首次尝试。
- 09:07 恢复检查依据实际内容而不是仅依据文件存在性。
- 当天内容未变化时会自动重新生成；只缺正式总结时只执行升级。
- 线上数据不再包含通用占位摘要。
- DeepSeek 不可用时仍有可追溯的论文原文摘录；恢复后自动升级。
- 任何失败不会覆盖上一份通过校验的数据。
